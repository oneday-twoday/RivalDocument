05 — Q&A

예상 면접 Q&A 21문항

Q01 NOL:EUM에서 Kinect 제스처를 직접 구현한 이유와 기술적 도전은? +
  • Kinect SDK 기본 제스처로는 비석치기·쥐불놀이 동작 인식이 불가능했습니다.
  • 비석치기는 손·어깨·골반 3관절 궤적을 분석해 투척 조건(이동 80cm / 0.17s)을 직접 설계했습니다.
  • 쥐불놀이는 Vector3.SignedAngle로 방향 벡터 변화를 감지해 회전 판별을 구현했습니다.
  • 핵심은 '사용자 체험이 자연스럽게 느껴지는 임계값'을 반복 테스트로 찾아낸 것입니다.
Q02 Ark with AI의 AI 파이프라인을 설계한 과정을 설명해주세요. +
  • STT(Whisper) → GPT Chat → TTS(OpenAI)의 전체 흐름을 VoiceCycleWithOpenAI 단일 스크립트로 통합했습니다.
  • 비동기 처리 중 사용자가 재입력할 수 있도록 CancellationToken으로 진행 중인 AI 호출을 즉시 중단합니다.
  • Intent Tag 시스템을 만들어 AI 텍스트 응답에서 의도를 파싱 → UnityEvent를 자동 호출하는 구조를 설계했습니다.
  • 이를 통해 음성 → 게임 로직 연결이 기획 변경에도 코드 수정 없이 Inspector에서 가능해졌습니다.
Q03 셧다운(Shutdown)의 셧다운 게이지 시스템은 어떻게 구현했나요? +
  • 게이지 값에 따라 Low/Medium/High 3단계로 나뉘는 시각 연출 시스템을 만들었습니다.
  • 카메라 셰이크: Cinemachine Perlin Noise 프로파일의 Amplitude/Frequency를 단계별로 동적 변경.
  • 플랫폼 노이즈: Mathf.PerlinNoise로 최소~최대 속도 사이 랜덤 진동, PingPong으로 알파 반복.
  • 비네트: Global Volume의 Vignette Intensity를 두 값 사이에서 왕복시켜 긴박감을 표현했습니다.
Q04 인터랙티브 전시 콘텐츠 개발 시 가장 중요하게 고려하는 점은 무엇인가요? +
  • 체험자의 '첫 3초'입니다. 설명 없이도 무엇을 해야 하는지 직관적으로 알 수 있어야 합니다.
  • NOL:EUM에서 놀이 도구를 향해 손을 움직이면 자동으로 선택되도록 한 것이 그 예입니다.
  • 연령·신체 조건이 다양한 관람객을 고려해 인식 임계값을 넉넉하게 설정하고 피드백을 명확히 주었습니다.
  • 기술적 완성도보다 '체험의 재미와 이해 가능성'을 우선했습니다.
Q05 Kinect를 사용하면서 겪은 가장 큰 문제와 해결 방법은? +
  • 비석치기의 '팔을 뒤로 젖히는 예비 동작' 문제였습니다.
  • 손 z축이 0 미만일 때(몸 뒤쪽)는 돌이 날아가지 않도록 조건을 추가했고,
  • 쥐불놀이에서는 팔 방향을 갑자기 바꿀 때 HoldTime을 두어 잠시 기존 방향을 유지하게 했습니다.
  • 실제 전시 현장 테스트를 통해 임계값을 반복 조정한 것이 가장 큰 해결책이었습니다.
Q06 여우의 밤을 Google Play에 출시한 경험에서 배운 점은? +
  • 빌드·배포 파이프라인 전 과정을 직접 경험했습니다: 서명 키스토어 생성, AAB 빌드, 스토어 메타데이터 작성.
  • 레벨 디자인 관점에서는 200개 장애물 데이터를 shuffleList로 관리하며 데이터 기반 콘텐츠 설계를 익혔습니다.
  • 실사용자 피드백을 통해 '직관적인 원터치 조작'의 중요성을 실감했고, 이후 프로젝트 설계에 반영했습니다.
Q07 플레이어 상태 관리를 Animator 대신 GameObject 활성화로 처리한 이유는? +
  • 각 상태별 컴포넌트 구성이 달라지는 상황에서 Animator는 시각적 전환만 담당해 로직 분리가 어려웠습니다.
  • GameObject ON/OFF 방식은 상태별 오브젝트를 에디터에서 직관적으로 확인·수정할 수 있어 팀 협업에 유리했습니다.
  • Enum 기반 상태 머신과 결합해 상태 전환 시 이전 상태 컴포넌트를 명시적으로 비활성화하므로 버그 추적도 용이했습니다.
Q08 전시 콘텐츠에서 UI/UX 설계 시 어떤 기준으로 접근하나요? +
  • 전시 환경 특성상 '튜토리얼 없이 1회 체험'이 기본 전제입니다.
  • NOL:EUM의 경우 한국어·영어 안내 텍스트를 동시 표기하고, 놀이 도구 오브젝트를 향한 손 움직임 하나로 선택되게 했습니다.
  • 체험 시간이 너무 길어지지 않도록 쥐불놀이에 도화선 슬라이더로 제한 시간을 시각화했습니다.
  • 핵심 원칙: 적은 입력으로 큰 시각 반응, 명확한 성공/실패 피드백.
Q09 팀 프로젝트에서 개발자로서 어떤 역할을 담당했나요? +
  • Shutdown(4인 팀)에서는 프로그래밍 리드로 아키텍처 설계와 핵심 시스템(게이지·플레이어·컷신)을 담당했습니다.
  • 비개발 팀원(디자이너·기획자)이 Inspector에서 값을 조정할 수 있도록 공개 변수와 이벤트를 적극 활용했습니다.
  • NOL:EUM에서는 Kinect 인식 로직 전담과 함께 Cinemachine 연출 전반을 맡아 기술-연출 연결 역할을 했습니다.
  • 기술과 기획 사이의 언어를 번역하는 '브릿지' 역할을 선호합니다.
Q10 인터랙티브 콘텐츠 회사에 지원한 이유와 본인의 강점은? +
  • 저는 게임 개발과 인터랙티브 미디어 아트 모두를 공부해온 개발자입니다.
  • NOL:EUM처럼 관람객이 몸으로 문화를 체험하는 콘텐츠를 만들 때 '기술'과 '감성' 둘 다 설계할 수 있다는 것이 강점입니다.
  • VR(Ark with AI)·미디어 월(NOL:EUM)·모바일(Shutdown, 여우의밤) 등 다양한 플랫폼 경험이 있습니다.
  • 전시 미디어 콘텐츠는 제가 가장 잘 할 수 있고, 가장 하고 싶은 분야입니다.
Q11 구현한 것 중 가장 기술적으로 도전적이었던 것은 무엇인가요? +
  • Ark with AI의 AI 파이프라인 설계입니다. 단순 API 호출이 아니라 VR 공간에 AI를 '캐릭터'로 통합하는 것이 목표였습니다.
  • 비동기 처리·CancellationToken·Intent 시스템을 하나의 플로우로 묶는 설계에 가장 많은 고민을 했습니다.
  • 완성 후 체험자가 AI와 자연스럽게 대화하며 우주선을 수리하는 경험을 보았을 때 가장 큰 성취감을 느꼈습니다.
Q12 Unity 버전 선택 기준이 있나요? +
  • 프로젝트 성격에 따라 선택합니다.
  • 안정성이 중요한 팀 프로젝트(Shutdown)는 LTS 버전(6000.0.60f1)을 사용했고,
  • 최신 XR 기능이 필요한 VR 프로젝트(Ark with AI)는 최신 버전(6000.3.6f1)을 택했습니다.
  • 현업에서는 프로젝트 요구사항과 팀 환경에 맞춰 유연하게 대응할 준비가 되어 있습니다.
Q13 5년 후 어떤 개발자가 되고 싶나요? +
  • 기술과 경험이 쌓인 인터랙티브 콘텐츠 전문가가 되고 싶습니다.
  • 단순한 구현 담당자를 넘어, 기획 단계부터 '어떤 인터랙션이 관람객에게 감동을 줄 수 있는가'를 제안할 수 있는 개발자가 목표입니다.
  • 장기적으로는 인터랙티브 미디어 아트와 게임 기술을 결합한 독자적인 콘텐츠를 제작하는 것이 꿈입니다.
Q14 async/await를 사용했는데, Coroutine으로도 가능하지 않나요? +
  • 가능하지만 async/await가 STT → Chat → TTS 순차 비동기 체인에 더 적합했습니다.
  • Coroutine은 yield return을 중첩해야 해서 가독성이 떨어지고, await로 쓰면 동기 코드처럼 읽히면서 예외처리도 try/catch로 깔끔하게 됩니다.
  • VR 환경에서 프레임 블로킹이 없어야 한다는 비동기 요구사항도 같이 충족됩니다.
Q15 CancellationToken은 어떤 상황에서 필요했나요? +
  • TTS 재생 중 사용자가 버튼을 다시 누르는 상황입니다.
  • speakCts.Cancel()을 호출하면 SpeakAsync 내부의 while(audioSource.isPlaying) 루프가 즉시 탈출해 기존 음성을 멈추고 새 녹음을 시작합니다.
  • 없었다면 이전 TTS가 끝날 때까지 기다려야 했고 VR 몰입이 크게 깨졌을 겁니다.
Q16 AI가 항상 현재 위치를 아는 건 어떻게 구현했나요? +
  • 대화 히스토리에 위치를 저장하지 않고, SendMessageAsync 호출 시마다 GetSystemContext()로 현재 위치와 방 상태를 새로 생성해 system 메시지로 주입합니다.
  • 히스토리에 넣으면 과거 위치 정보가 누적돼 혼란이 생기고 토큰도 낭비됩니다.
  • 매번 갱신하는 방식이 더 정확하고 효율적입니다.
Q17 Intent 시스템에서 왜 방(Room) 단위로 분리했나요? +
  • 같은 태그명이 방마다 다른 이벤트를 실행해야 하는 경우가 생길 수 있습니다.
  • 플레이어가 없는 방의 이벤트가 실수로 실행되는 오발동을 방지하기 위해 방 열림 여부 + 플레이어 위치를 동시에 검사하는 2중 가드를 뒀습니다.
Q18 시스템 프롬프트 설계에서 가장 신경 쓴 부분은 무엇인가요? +
  • AI가 단순 답변 봇이 아니라 "손상된 우주선의 AI 캐릭터"처럼 느껴지도록 페르소나를 잡는 것이었습니다.
  • 기계적인 말투 대신 감정 있는 어조, 과도한 반복 없이 상황의 긴박함을 전달하는 답변 간결성을 설계했습니다.
  • 3단계 조력 방식(관찰 → 유도 → 제안)으로 힌트를 줘서 체험자가 스스로 해결하는 느낌을 살렸습니다.
Q19 ChatGPT_OpenAi.cs와 OpenAI_Chat.cs가 따로 있는 이유는? +
  • ChatGPT_OpenAi.cs는 초기 프로토타입 단계에서 빠르게 API 연결을 테스트하던 코드입니다.
  • 이후 RapidFramework 네임스페이스 기반으로 구조를 재설계하면서 OpenAI_Chat.cs로 완전히 새로 작성했습니다.
  • 프로토타입과 실제 구현을 분리해서 관리한 흔적입니다. 이런 과정이 오히려 개선 의지를 보여준다고 생각합니다.
Q20 API 키 관리를 ScriptableObject로 한 이유는? +
  • 코드에 하드코딩하면 Git에 올라갔을 때 키가 노출됩니다.
  • ScriptableObject(OpenAI_KeySetting)로 분리하면 .gitignore에 해당 에셋만 추가해 키를 안전하게 관리할 수 있습니다.
  • 팀원이 각자 키를 Inspector에서 설정할 수 있어 협업에도 유리합니다.
Q21 TTS에 글리치 효과를 넣은 이유가 있나요? +
  • 순수 기술 이유가 아닌 연출 이유입니다. 우주선이 손상된 세계관에서 AI 음성이 너무 깨끗하면 몰입이 깨집니다.
  • 8% 확률로 pitch를 미세하게 흔드는 것만으로도 "고장난 AI"라는 느낌이 생겨 체험자가 상황에 더 몰입하게 됩니다.
  • 기술과 연출을 함께 고려한 디테일입니다. 이런 작은 연출 판단이 전시 콘텐츠의 품질을 가릅니다.

🏢 알케미스타 회사 분석

팀 구조 — 프로그램 솔루션팀 / 연출·디자인팀

딱 두 팀으로 구성된 소기업 구조. 개발자는 디자이너·연출자와 직접 붙어서 일한다. 기술과 연출 사이 소통 능력이 실질적으로 중요한 환경.

내 Inspector 기반 설계, 팀원이 값 조정 가능한 아키텍처가 이 구조에 정확히 맞는다.
기술 포트폴리오 — Kinect 핵심 + VR 성장 중

채용공고 나열 순서: 터치스크린 → AR → Kinect → VR. VR은 "최근 활발히"라고 별도 강조 → 성장 중인 신규 영역. Kinect로 안정 납품, VR로 확장하는 전환점에 내가 들어가는 것.

면접 분위기 — 캐주얼하고 실무 중심

"창의적이고 즐겁게, 자유롭게"라는 자기 표현 → 수평적 분위기 암시. 대표나 팀장이 직접 진행하는 포트폴리오 중심 대화형 면접일 가능성이 높다.

딱딱하게 외운 답변보다 자연스럽게 프로젝트 얘기를 꺼내는 것이 더 잘 맞는 환경.
공개 면접 후기 없음 → 오히려 유리

잡플래닛·사람인 등 주요 플랫폼에 면접 후기가 단 한 건도 없다. 지원자가 드문 소기업이라는 의미 → 충분히 준비된 지원자에게 희소성이 생긴다.
